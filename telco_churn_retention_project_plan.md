# 통신사 고객 이탈 예측 및 자동 리텐션 에이전트 프로젝트 계획서

> Telco Customer Churn Prediction & Automated Retention Agent
> 약 2개월(8주) 실전 모의 프로젝트 / 머신러닝·딥러닝·생성형 AI·LLM 엔지니어링·모델 서빙·AI Agent 통합
> 데이터셋: Kaggle Telco Customer Churn (`blastchar/telco-customer-churn`)

## 1. 프로젝트 개요

### 1) 프로젝트 주제명

**통신사 고객 이탈(Churn) 예측 모델 기반 자동 리텐션(고객 유지) 에이전트 개발**

한 문장으로 요약하면 다음과 같습니다.

> "떠날 것 같은 고객을 미리 찾아내고(예측), 그 고객에게 맞는 메시지와 혜택을 자동으로 제안하는(처방) AI 시스템을 만든다."

이 프로젝트의 핵심은 **예측에서 멈추지 않고 행동(액션)까지 자동화**한다는 점입니다. 단순 분류 모델 과제와 차별화되는 지점이며, "그래서 이 모델로 무엇을 했는가?"라는 면접 질문에 명확히 답할 수 있는 구조입니다.

### 2) 프로젝트 수행 배경

**(가) 비즈니스 배경**

- 통신 시장은 포화 상태라 신규 고객을 늘리기 어렵습니다. 따라서 **기존 고객을 지키는 일(Retention)**이 매출 방어의 핵심입니다.
- 일반적으로 신규 고객 한 명을 데려오는 비용이 기존 고객을 유지하는 비용보다 훨씬 큽니다. 이탈을 조금만 줄여도 수익에 큰 영향을 줍니다.
- 문제는 이탈을 **사후에** 알면 늦다는 점입니다. 고객이 해지한 뒤에는 되돌리기 어렵습니다.

**(나) 문제 정의 (Problem)**

1. 어떤 고객이 곧 이탈할지 **사전에** 알 수 없다.
2. 안다고 해도, **누구에게 / 어떤 메시지로 / 어떤 혜택을** 제안할지 사람이 일일이 판단하기 어렵다.
3. 대상 고객이 많아지면 **수작업으로는 감당이 불가능**하다.

**(다) 가치 정의 (Value)**

| 단계 | 질문 | 담당 기술 |
|---|---|---|
| 예측 | 누가 떠날까? | 머신러닝 / 딥러닝 |
| 해석 | 왜 떠날까? | 설명가능 AI(SHAP) |
| 처방 | 어떻게 잡을까? | 생성형 AI(LLM) |
| 실행 | 자동으로 무엇을 할까? | AI Agent |
| 운영 | 잘 되고 있나? | 모델 서빙 & 대시보드 |

즉 **"예측(Prediction)"과 "처방(Prescription)"을 하나의 파이프라인으로 연결**하는 것이 이 프로젝트의 가치입니다.

**(라) 학습 측면 배경**

수강한 전 과목(ML, DL, 생성형 AI/LLM, LLM 엔지니어링, 모델 서빙·대시보드, AI Agent)을 **하나의 흐름 안에서** 사용하므로, 개별 기술이 실제 제품에서 어떻게 맞물리는지 체득할 수 있습니다.

### 3) 프로젝트 목표 및 수행 내용

**(가) 목표 (측정 가능하게)**

정량 목표(예시 기준이며 데이터 분석 후 조정)

- 이탈 예측 모델 ROC-AUC 0.83 이상, PR-AUC 0.63 이상
- 이탈 고객 재현율(Recall) 0.75 이상 (떠날 고객을 놓치지 않는 것이 더 중요)
- 위험 고객 1건당 맞춤 리텐션 메시지 자동 생성 100% 자동화
- 위험도 구간별 액션 자동 분기 규칙 구현

정성 목표

- API 키 없이도 전체 데모가 동작(시뮬레이션 모드)

- 비개발자도 이해할 수 있는 대시보드 제공

**(다) 데이터 사용 시 주의점 (학생들이 자주 막히는 부분)**

1. `TotalCharges`는 숫자처럼 보이지만 **공백 문자열 11건**이 섞여 있어 그대로 읽으면 문자열로 인식됩니다. 숫자로 변환 후 결측 처리가 필요합니다.
   - 참고: 이 11건은 모두 `tenure`(가입기간)가 0인 **신규 고객**입니다. 아직 누적 요금이 없어 공백인 것이므로, 0 또는 중앙값으로 대체하는 근거가 명확합니다.
2. `Churn`은 Yes/No 문자열이므로 1/0으로 변환해야 합니다.
3. 이탈 비율이 약 **26.5%**로 불균형합니다. 정확도(Accuracy)만 보면 안 되고 재현율·PR-AUC를 함께 봐야 합니다.
4. `SeniorCitizen`은 이미 0/1 정수이므로 추가 인코딩이 불필요합니다.

## 2. 상세 구현/분석 방법 및 예시

### 1) 구현/분석 프로세스

각 모듈을 "문제·가치정의 → 분석(EDA) → 설계(인터페이스/테스트 정의) → TDD 개발"의 순서로 진행하고, 다음 모듈로 넘어갑니다. 별도의 고정된 스프린트 반복 없이, 모듈(M1~M7) 단위로 순차 진행하며 필요한 경우에만 앞 단계로 되돌아가 보완합니다.

```
문제·가치정의 ──▶ 분석(EDA) ──▶ 설계(인터페이스/테스트 정의) ──▶ TDD 개발
     ▲                                                              │
     └──────────── (필요 시) 앞 단계로 피드백 반영 ◀────────────────┘
```

전체 데이터 흐름(엔드투엔드)은 다음과 같습니다.

```
고객 데이터(CSV)
      │
      ▼
 [전처리/피처]  ──▶  [이탈 예측 모델]  ──▶  이탈 확률(0~1)
                                              │
                          ┌───────────────────┘
                          ▼
                    [SHAP 사유 해석]  ──▶  "왜 위험한가" 근거
                          │
                          ▼
                  [리텐션 에이전트] ── 위험도 분기 ──┐
                          │                          │
              위험 높음   │   위험 중간   │  위험 낮음
                ▼         ▼              ▼
        [LLM 맞춤 메시지] [상담 연결]   [관찰만]
                          │
                          ▼
                 [대시보드 / 로그 기록]
```

### 2) 구현/분석 방법론

**(가) 분석 방법론: CRISP-DM 매핑**

| CRISP-DM 단계 | 이 프로젝트에서 하는 일 |
|---|---|
| 비즈니스 이해 | 이탈의 비즈니스 비용 정의, 목표 지표 설정 |
| 데이터 이해 | EDA, 분포·상관·결측 파악 |
| 데이터 준비 | 결측 처리, 인코딩, 피처 엔지니어링 |
| 모델링 | ML/DL 비교, 불균형 처리, 임계값 튜닝 |
| 평가 | Recall/PR-AUC, 비즈니스 비용 기반 평가 |
| 배포 | API 서빙 + 에이전트 + 대시보드 |

**(나) 불균형 데이터 처리 방법론**

- 클래스 가중치 부여를 1순위로 사용(가장 간단하고 안전).
- 임계값 튜닝: 기본 0.5가 아니라 "이탈 고객을 놓치는 비용"을 반영해 임계값을 낮춤.
- 오버샘플링(SMOTE 등)은 보조적으로 실험(과적합 주의, 반드시 학습 데이터에만 적용).

**(다) 평가 지표 선택 근거**

이탈 고객(소수)을 놓치는 것이 더 큰 손해이므로 **Recall과 PR-AUC를 주지표**로, ROC-AUC는 보조로 봅니다. 비즈니스적으로는 "리텐션 비용 대비 방어 매출"로 임계값을 정합니다.

**(라) 설명가능성(SHAP)**

모델이 왜 그 고객을 위험하다고 봤는지 SHAP 값으로 설명합니다. 이 사유는 그대로 LLM 메시지 생성의 입력이 됩니다(예측과 처방의 연결고리).

**(마) LLM 프롬프트 설계 원칙**

- 입력: 고객 특징 + 이탈 사유(SHAP 상위 요인) + 위험도
- 출력: 정중하고 구체적인 리텐션 메시지(혜택 제안 포함)
- 가드레일: 과장/허위 혜택 금지, 사실 기반, 톤 일관

**(바) 에이전트 의사결정 규칙(초기 버전은 규칙 기반)**

| 위험도 구간 | 등급 | 액션 |
|---|---|---|
| 0.7 이상 | 고위험 | LLM 맞춤 메시지 생성 + 프리미엄 혜택 제안 |
| 0.4 ~ 0.7 | 중위험 | 상담사 연결 추천 + 표준 혜택 |
| 0.4 미만 | 저위험 | 별도 액션 없이 관찰 |

**(사) TDD 방법**

테스트를 먼저 정의하고(Red) → 통과하는 최소 구현(Green) → 정리(Refactor)합니다. 전처리, 임계값 분기, 에이전트 규칙처럼 **입출력이 명확한 부분**부터 테스트로 보호합니다.

**(아) 공정성/편향(Fairness) 점검**

이탈 모델이 특정 인구통계 집단에 체계적으로 불리하게 작동하면, 리텐션 혜택이 불공정하게 배분될 수 있습니다.

- 민감 속성(예: `gender`, `SeniorCitizen`)별로 **재현율·정밀도·이탈 예측률 격차**를 측정합니다.
- 격차가 크면 임계값을 집단별로 점검하거나, 해당 속성을 입력에서 제외하는 실험을 합니다(다만 제외해도 상관 피처로 정보가 누수될 수 있음을 함께 분석).
- 발표 시 "모델이 공정한지 확인했다"는 점은 신뢰성 어필 포인트가 됩니다.

**(자) LLM 메시지 품질 평가**

생성된 리텐션 메시지는 주관 평가로 끝내지 않고, 가벼운 평가 루브릭으로 검증합니다.

| 평가 항목 | 확인 방법 |
|---|---|
| 가드레일 준수 | 금지어/허위 혜택 문구 미포함 자동 점검 |
| 사유 반영 | SHAP 상위 사유가 메시지에 반영되었는지 확인 |
| 톤/형식 | 정중체·길이 제한·필수 요소(혜택 제안) 포함 여부 |
| 사실성(선택) | LLM-as-a-judge로 사실 일치 여부 보조 채점 |

**(차) 에이전트 고도화 (스트레치 목표 — AI Agent 과목 연계)**

초기 버전은 규칙 기반((바))으로 빠르게 완성하되, 시간이 남으면 **LLM 기반 에이전트(툴 호출)**로 확장합니다.

- 위험도 조회·사유 조회·메시지 생성·상담 연결을 각각 **툴(tool)**로 정의하고, LLM이 상황에 따라 호출 순서를 스스로 결정하게 합니다.
- 규칙 기반 대비 "왜 에이전트가 필요한가"를 보여주는 비교 데모가 되어 발표 임팩트가 큽니다.

### 3) 구현/분석 상세 내용 및 예시

각 모듈의 목적·입력·처리·출력과 핵심 설계 포인트를 정리합니다. (수치는 설명용 예시이며 실제 분석 후 달라질 수 있습니다.)

#### [M1] 데이터 로드·검증 및 전처리

| 항목 | 내용 |
|---|---|
| 목적 | 원본 CSV를 안전하게 적재하고 모델 입력 형태로 정제 |
| 입력 | 원본 CSV (7,043행 × 21열) |
| 처리 | `TotalCharges` 숫자 변환·결측 처리, `Churn` 1/0 변환, 수치형 결측 대체·표준화, 범주형 인코딩, 학습/검증 분할(stratify) |
| 출력 | 전처리된 학습/검증 데이터셋 |
| 핵심 포인트 | 전처리를 하나의 파이프라인으로 묶어 데이터 누수 차단, 분할 시 클래스 비율 유지 |

> 예시(목표 산출): 타깃 분포 약 73.5% : 26.5%, `TotalCharges` 공백 11건(모두 신규 고객)을 결측 처리.

#### [M2] 이탈 예측 모델 학습·평가

| 항목 | 내용 |
|---|---|
| 목적 | 고객별 이탈 위험도(0~1) 산출 모델 구축 |
| 입력 | 전처리된 학습 데이터 |
| 처리 | 전통 ML(예: XGBoost)과 딥러닝(Tabular) 비교, 불균형 보정(클래스 가중치), 임계값 튜닝, 모델 저장 |
| 출력 | 학습된 모델 파일, 성능 리포트 |
| 핵심 포인트 | 정확도가 아닌 Recall·PR-AUC 중심 평가, 이탈 누락 최소화를 위해 임계값을 0.5보다 낮춤 |

> 예시(목표 성능): ROC-AUC 0.83+, PR-AUC 0.63+, Recall 0.75+ (임계값 조정 적용).

#### [M3] 이탈 사유 해석 (SHAP)

| 항목 | 내용 |
|---|---|
| 목적 | "왜 이 고객이 위험한가"를 사람이 읽을 수 있게 설명 |
| 입력 | 학습된 모델 + 특정 고객 데이터 |
| 처리 | SHAP 기여도 계산 → 상위 위험 요인·보호 요인 정리 |
| 출력 | 위험/보호 요인 목록(기여도 포함) |
| 핵심 포인트 | 이 사유가 곧 LLM 메시지 생성의 입력이 됨(예측·처방 연결) |

> 예시(개념): 한 고위험 고객(이탈 확률 0.82)의 주요 위험 요인 — 월 단위 계약, 짧은 가입기간, 광인터넷 사용 / 보호 요인 — 기술지원 가입.

#### [M4] 리텐션 메시지 생성 (LLM, 시뮬레이션 모드 지원)

| 항목 | 내용 |
|---|---|
| 목적 | 고객·사유·위험도에 맞춘 정중한 리텐션 메시지 자동 작성 |
| 입력 | 고객 특징 + SHAP 상위 사유 + 위험도 |
| 처리 | 프롬프트 구성 → LLM 호출(키 없으면 템플릿 기반 시뮬레이션으로 폴백) |
| 출력 | 맞춤 리텐션 메시지 텍스트 |
| 핵심 포인트 | API 키 없이도 전체 데모가 동작하도록 시뮬레이션 모드 제공, 가드레일(허위·과장 혜택 금지) 준수 |

> 예시(개념): "약정 시 월 요금 15% 할인과 기술지원 무료 혜택을 제안드립니다…" 형태의 정중한 한국어 메시지.

#### [M5] 리텐션 에이전트 (의사결정)

| 항목 | 내용 |
|---|---|
| 목적 | 위험도에 따라 적절한 리텐션 액션을 자동 결정 |
| 입력 | 고객 위험도 + 사유 |
| 처리 | 위험 구간 판정(고/중/저) → 액션 분기, 고위험은 메시지 생성 연계 |
| 출력 | 액션(메시지 발송 / 상담 연결 / 관찰) + (해당 시) 생성 메시지 |
| 핵심 포인트 | 초기엔 규칙 기반, 스트레치로 LLM 툴 호출 에이전트로 확장 가능 |

> 예시(개념): 위험도 0.82 → 메시지 발송, 0.52 → 상담사 연결, 0.18 → 관찰.

#### [M6] 서빙 & 대시보드

- **추론 API**: 고객 정보를 받아 `{이탈확률, 사유, 액션, 메시지}`를 반환. 학습 단계에서 저장한 모델을 로드해 사용.
- **대시보드**: 전체 이탈 위험 분포, 고위험 고객 리스트, 액션 현황, 모델 성능 추이를 시각화.

```
[대시보드 화면 구성 예시]
┌───────────────────────────────────────────────┐
│ 전체 이탈 위험 분포   |  이번 주 고위험 고객 수 │
├───────────────────────────────────────────────┤
│ 고위험 고객 테이블 (ID / 위험도 / 사유 / 액션) │
├───────────────────────────────────────────────┤
│ 액션 현황(발송/상담연결/관찰)  | 모델 성능 추이 │
└───────────────────────────────────────────────┘
```

#### [M7] 평가 & 테스트(TDD)

- 입출력이 명확한 핵심 로직(데이터 로드·전처리·임계값 분기·에이전트 규칙)을 자동화된 테스트로 보호합니다.
- 성능 임계값 회귀 테스트로 모델 품질 저하를 조기에 감지하고, 통합 단계에서는 엔드투엔드 회귀 테스트로 전체 흐름을 검증합니다.

### 4) 데이터 리소스 활용 상세

**(가) 컬럼별 활용 전략**

| 활용 목적 | 사용 컬럼 | 방법 |
|---|---|---|
| 예측 입력 | tenure, MonthlyCharges, Contract 등 | 인코딩/스케일링 후 모델 입력 |
| 파생 변수 | 가입 서비스들 | 가입 서비스 개수(num_services) 합산 |
| 파생 변수 | tenure, TotalCharges | 평균 월 지출 = TotalCharges / tenure |
| 위험 해석 | 전체 피처 | SHAP 기여도 |
| 메시지 생성 | SHAP 상위 사유 + 계약/요금 | LLM 프롬프트 입력 |

**(나) 데이터 누수(leakage) 방지 원칙**

- 전처리(인코딩·스케일링·결측대체)는 반드시 **학습 데이터로만 적합(fit)**하고 검증/테스트에는 적용(transform)만 합니다. 전처리를 파이프라인 안에 포함하면 구조적으로 보호됩니다.
- 타깃(Churn)에서 파생된 정보를 입력 피처로 넣지 않습니다.
- `customerID`는 식별자이므로 학습 피처에서 제외합니다.

**(다) 파생 변수 설계**

- **가입 서비스 개수(num_services)**: 부가 서비스 가입 수를 합산해 충성도·연계 정도를 나타내는 지표로 활용.
- **평균 월 지출(avg_monthly_spend)**: 누적 요금을 가입기간으로 나눈 값. 가입기간이 0인 신규 고객은 분모 처리에 유의(0 → 1로 치환 등).

**(라) 데이터 버전 관리**

- 원본 데이터는 절대 수정하지 않고, 전처리 결과만 별도 산출물로 저장합니다.
- 전처리 규칙이 바뀌면 결과 파일명에 버전을 붙여 이력을 관리합니다.

### 5) 예상 계획 및 일정

8주 기준이며, 주차별로 핵심 목표와 산출물을 두고 모듈(M1~M7) 단위로 순차 진행합니다.

| 주차 | 핵심 목표 | 주요 산출물 | TDD 체크포인트 |
|---|---|---|---|
| 1주 | 문제·가치정의, 데이터 이해(EDA) | 계획서, EDA 결과 | 데이터 로드/스키마 검증 테스트 |
| 2주 | 전처리·피처 엔지니어링 | 전처리 모듈 | 전처리 입출력 테스트 |
| 3주 | 베이스라인 ML 모델 | 분류 모델, 평가 리포트 | 성능 임계값 회귀 테스트 |
| 4주 | 모델 고도화(불균형·임계값·DL 비교) | 최종 모델, 비교표 | 임계값 분기 테스트 |
| 5주 | SHAP 해석 + LLM 메시지 생성 | 해석 모듈, 메시지 생성 모듈 | 프롬프트 구성 테스트 |
| 6주 | 리텐션 에이전트 통합 | 에이전트 모듈 | 액션 분기 규칙 테스트 |
| 7주 | 서빙(API) + 대시보드 | 추론 API, 대시보드 | API 응답 스키마 테스트 |
| 8주 | 통합·시연·발표 준비 | 데모, 발표자료, README | 전체 회귀 테스트(E2E) |

**(가) 역할 분담 예시 (4인 팀 가정)**

- 데이터/모델 담당: M1~M2
- 해석/LLM 담당: M3~M4
- 에이전트/통합 담당: M5
- 서빙/대시보드 담당: M6 (테스트(M7)는 전원 공동)

**(나) 리스크 및 대응**

| 리스크 | 대응 |
|---|---|
| 불균형으로 성능이 낮게 보임 | Recall/PR-AUC 중심 평가, 임계값 튜닝 |
| LLM API 비용/키 부재 | 시뮬레이션 모드로 전체 데모 동작 보장 |
| 일정 지연 | M1~M6을 우선 완성, 서빙/대시보드는 최소기능(MVP)으로 |
| 데이터 누수 | 전처리를 파이프라인에 포함해 구조적으로 차단 |
| 라이브러리 호환 오류(예: Python 3.14) | 3.11/3.12 가상환경 + 버전 고정으로 팀 환경 통일 |
| 모델 편향(특정 집단 불이익) | 집단별 지표 격차 측정·리포트(공정성 점검) |
| LLM 메시지 품질 편차 | 가드레일 자동 점검 + 평가 루브릭 적용 |
| 실험 재현 불가/결과 비교 어려움 | 난수 시드 고정, 실험 기록(파라미터·지표) 관리 |

## 부록: 발표 시 강조 포인트

1. "예측만 한 게 아니라 **행동까지 자동화**했다" — 예측·해석·생성·에이전트의 연결을 보여줍니다.
2. "왜 위험한지 설명할 수 있다" — SHAP 기반 사유 제시로 신뢰성을 강조합니다.
3. "비용을 고려해 임계값을 정했다" — 비즈니스 관점의 평가를 어필합니다.
4. "테스트로 보호했다" — TDD로 안정성을 확보했음을 보여줍니다.
5. "공정성까지 점검했다" — 집단별 성능 격차 확인으로 책임감 있는 AI를 어필합니다.
