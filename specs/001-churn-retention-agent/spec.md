## User Scenarios & Testing *(mandatory)*

이 시스템의 1차 사용자는 **통신사 리텐션(고객 유지) 운영 담당자**이며, 2차 사용자는 시스템을 구축·운영하는 **데이터/ML 팀(수강생)**입니다. 아래 사용자 스토리는 가치 순으로 우선순위를 매겼고, 각각 독립적으로 시연 가능한 단위입니다.

### User Story 1 - 이탈 위험 고객 식별 (Priority: P1)

리텐션 담당자는 매주 고객 명단에 대해 "곧 이탈할 가능성이 높은 고객"을 위험도 점수와 함께 랭킹 형태로 받아본다. 담당자는 한정된 시간과 예산 안에서 누구에게 먼저 개입할지 결정할 수 있다.

**Why this priority**: 예측이 없으면 나머지 모든 단계(해석·메시지·액션)가 성립하지 않는다. 위험 랭킹 하나만으로도 "수작업 전수 대응"을 "고위험 우선 대응"으로 바꾸는 즉각적 가치를 제공하므로 단독 MVP가 된다.

**Independent Test**: 과거 데이터로 학습한 모델에 보류 고객 집합을 입력해 위험도 랭킹을 산출하고, 상위 구간이 실제 이탈 고객을 얼마나 포착하는지(리프트/재현율)로 단독 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 학습이 완료된 모델과 평가용 고객 집합이 주어졌을 때, **When** 담당자가 위험 랭킹을 요청하면, **Then** 각 고객에 대해 0~1 사이의 이탈 위험도와 위험 등급(고/중/저)이 부여된 정렬된 목록을 받는다.
2. **Given** 신규 고객(가입기간 0개월, 누적요금 공백)이 입력되었을 때, **When** 위험도를 산출하면, **Then** 오류 없이 합리적인 위험도가 계산된다(결측이 사전 정의된 규칙으로 처리됨).
3. **Given** 이탈 비율이 약 26.5%로 불균형한 데이터일 때, **When** 모델 성능을 평가하면, **Then** 정확도가 아닌 재현율·PR-AUC 기준으로 성능이 보고된다.

---

### User Story 2 - 이탈 사유 설명 (Priority: P2)

담당자는 특정 고객이 "왜" 위험한지에 대한 핵심 사유(상위 위험 요인과 보호 요인)를 사람이 읽을 수 있는 형태로 확인한다. 단순 점수가 아니라 근거가 있어 신뢰하고 행동할 수 있다.

**Why this priority**: 사유 없는 점수는 현장에서 신뢰받지 못하고, 다음 단계인 맞춤 메시지 생성의 입력이 된다. 예측(P1)을 처방으로 잇는 연결고리다.

**Independent Test**: 임의의 고위험 고객 한 명에 대해 사유 추출을 실행해, 모델이 사용한 피처 기여도 상위 요인이 일관되게 반환되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 위험도가 산출된 고객이 있을 때, **When** 사유 설명을 요청하면, **Then** 위험을 높인 상위 요인과 위험을 낮춘 보호 요인이 기여도와 함께 반환된다.
2. **Given** 동일한 고객·동일한 모델일 때, **When** 사유 설명을 반복 실행하면, **Then** 동일한 상위 요인이 일관되게 반환된다(재현 가능).

---

### User Story 3 - 맞춤 리텐션 액션 자동 결정·생성 (Priority: P3)

담당자는 고위험 고객에 대해 자동으로 결정된 리텐션 액션(맞춤 메시지 발송 / 상담사 연결 / 관찰)과, 발송 대상에게는 사유 기반의 정중한 맞춤 메시지를 함께 받는다. 외부 LLM API 키가 없어도 전체 흐름이 시뮬레이션 모드로 동작한다.

**Why this priority**: "예측에서 멈추지 않고 행동까지 자동화"라는 프로젝트의 차별점이자 핵심 가치다. 다만 P1·P2가 선행되어야 의미가 있으므로 세 번째 우선순위다.

**Independent Test**: 위험도와 사유를 입력으로 액션 분기와 메시지 생성을 실행해, 위험 구간별로 올바른 액션이 선택되고 메시지가 가드레일(허위·과장 혜택 금지)을 위반하지 않는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 위험도 0.7 이상 고객일 때, **When** 에이전트가 액션을 결정하면, **Then** "맞춤 메시지 발송" 액션이 선택되고 사유 기반 메시지가 생성된다.
2. **Given** 위험도 0.4~0.7 고객일 때, **When** 에이전트가 액션을 결정하면, **Then** "상담사 연결" 액션이 선택된다.
3. **Given** 위험도 0.4 미만 고객일 때, **When** 에이전트가 액션을 결정하면, **Then** "관찰" 액션이 선택되고 메시지는 생성되지 않는다.
4. **Given** 외부 LLM API 키가 설정되지 않았을 때, **When** 메시지 생성을 요청하면, **Then** 시뮬레이션 모드로 템플릿 기반 메시지가 정상 반환된다.

---

### User Story 4 - 운영 현황 대시보드 (Priority: P4)

담당자와 관리자는 전체 이탈 위험 분포, 고위험 고객 리스트, 액션 현황, 모델 성능 추이를 한 화면에서 모니터링한다.

**Why this priority**: 가치는 높지만 P1~P3의 결과를 소비하는 표현 계층이므로, 핵심 파이프라인 이후에 구축해도 무방하다.

**Independent Test**: 저장된 추론 결과 파일을 입력으로 대시보드를 띄워, 위험 분포·고위험 리스트·액션 집계가 정확히 렌더링되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 추론·액션 결과가 저장되어 있을 때, **When** 대시보드를 열면, **Then** 위험 분포, 고위험 고객 테이블(ID·위험도·사유·액션), 액션 집계가 표시된다.

---

### Edge Cases

- **결측·이상 입력**: `TotalCharges`가 공백인 신규 고객(가입기간 0) 입력 시에도 사전 정의된 결측 처리 규칙에 따라 오류 없이 처리된다.
- **알 수 없는 범주값**: 학습 시 보지 못한 새로운 범주값이 입력되면 모델은 실패하지 않고 안전하게 처리(무시/기본 인코딩)한다.
- **LLM 호출 실패/타임아웃**: 실제 API 모드에서 호출이 실패하면 시뮬레이션 템플릿으로 폴백하여 흐름이 끊기지 않는다.
- **전부 저위험인 배치**: 고위험 고객이 0명인 배치에서도 대시보드와 로그가 비어 있는 상태를 정상 표시한다.
- **공정성 경계**: 특정 인구통계 집단(성별, 고령자 여부)에 대해 위험도가 체계적으로 편향되지 않는지 점검 가능해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 고객 데이터 집합을 입력받아 각 고객에 대해 0~1 범위의 이탈 위험도를 산출해야 한다(MUST).
- **FR-002**: 시스템은 산출된 위험도를 기준으로 고객을 정렬하고 위험 등급(고/중/저)을 부여해야 한다(MUST).
- **FR-003**: 시스템은 불균형 데이터에서 소수 클래스(이탈) 포착을 우선하는 성능 지표(재현율, PR-AUC)로 모델을 평가·보고해야 한다(MUST).
- **FR-004**: 시스템은 결측·형 변환이 필요한 입력(예: 공백 `TotalCharges`, Yes/No 타깃)을 사전 정의된 규칙으로 처리해야 한다(MUST).
- **FR-005**: 시스템은 학습 데이터에서만 전처리 규칙을 적합(fit)하고 평가·운영 데이터에는 적용(transform)만 하여 데이터 누수를 구조적으로 차단해야 한다(MUST).
- **FR-006**: 시스템은 각 고객의 위험도에 대해 상위 위험 요인과 보호 요인을 기여도와 함께 사람이 읽을 수 있는 형태로 제공해야 한다(MUST).
- **FR-007**: 시스템은 위험도 구간에 따라 리텐션 액션(메시지 발송 / 상담사 연결 / 관찰)을 자동으로 결정해야 한다(MUST).
- **FR-008**: 시스템은 발송 대상 고객에 대해 사유와 위험도를 반영한 맞춤 리텐션 메시지를 자동 생성해야 한다(MUST).
- **FR-009**: 생성된 메시지는 허위·과장 혜택을 포함하지 않고 사실 기반·일관된 톤을 유지하는 가드레일을 준수해야 한다(MUST).
- **FR-010**: 시스템은 외부 LLM API 키 없이도 전체 흐름이 동작하는 시뮬레이션 모드를 제공해야 한다(MUST).
- **FR-011**: 시스템은 학습된 모델을 저장·로드하여 동일 입력에 대해 재현 가능한 추론을 제공해야 한다(MUST).
- **FR-012**: 시스템은 추론·액션 결과(고객ID, 위험도, 사유, 액션, 메시지)를 로그로 저장해야 한다(MUST).
- **FR-013**: 시스템은 위험 분포, 고위험 고객 리스트, 액션 현황, 모델 성능을 시각화하는 대시보드를 제공해야 한다(SHOULD).
- **FR-014**: 시스템은 추론 결과를 외부에서 호출 가능한 형태(요청→{위험도, 사유, 액션, 메시지})로 제공해야 한다(SHOULD).
- **FR-015**: 입출력이 명확한 핵심 로직(전처리, 임계값 분기, 에이전트 규칙)은 자동화된 테스트로 검증되어야 한다(MUST).
- **FR-016**: 시스템은 인구통계 집단(성별, 고령자 여부)별 위험도·성능 편향을 점검할 수 있는 수단을 제공해야 한다(SHOULD).

### Key Entities *(include if feature involves data)*

- **Customer(고객)**: 예측 대상 단위. 인구통계, 가입 서비스, 계약·결제, 요금 등의 속성을 가짐. 식별자(`customerID`)는 예측 입력에서 제외.
- **ChurnRiskScore(이탈 위험도)**: 고객별 0~1 위험 확률과 위험 등급. 임계값에 따라 등급이 결정됨.
- **RiskExplanation(위험 사유)**: 특정 고객의 위험도를 구성하는 상위 위험 요인·보호 요인과 그 기여도.
- **RetentionAction(리텐션 액션)**: 위험 등급에 따라 결정되는 액션(메시지 발송 / 상담사 연결 / 관찰).
- **RetentionMessage(리텐션 메시지)**: 발송 대상 고객을 위해 사유 기반으로 생성된 맞춤 텍스트.
- **ActionLog(액션 로그)**: 추론·액션 결과 기록(고객ID, 위험도, 사유, 액션, 메시지, 시점).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 위험도 상위 20% 구간이 실제 이탈 고객의 50% 이상을 포착한다(상위 구간 리프트 확보).
- **SC-002**: 이탈 고객 재현율 0.75 이상을 달성하면서, 운영 가능한 수준의 정밀도(과도한 헛개입 회피)를 유지한다.
- **SC-003**: 모든 고위험 고객에 대해 사유 설명과 리텐션 메시지가 100% 자동 생성된다(수작업 0건).
- **SC-004**: 외부 API 키 없이 전체 데모(데이터 적재→예측→해석→메시지→액션→대시보드)가 처음부터 끝까지 오류 없이 동작한다.
- **SC-005**: 위험도 구간별 액션 분기가 정의된 규칙과 100% 일치한다(경계값 포함).
- **SC-006**: 한 배치(수천 명 규모) 고객에 대한 위험도·액션 산출이 담당자가 기다릴 수 있는 시간(수십 초 이내)에 완료된다.
- **SC-007**: 인구통계 집단(성별, 고령자 여부)별 재현율 격차가 점검 리포트로 산출되어, 편향 존재 여부를 판단할 수 있다.

## Assumptions

- 데이터는 Kaggle Telco Customer Churn 단일 CSV(7,043행 × 21열)를 사용하며, 실시간 스트리밍이 아닌 배치(주기적) 처리 시나리오를 가정한다.
- 위험 등급 임계값(고위험 0.7, 중위험 0.4)과 추론 임계값은 초기 기본값이며 데이터 분석 후 비용 기반으로 조정 가능하다고 가정한다.
- 리텐션 메시지 발송·상담 연결의 실제 전송 채널(문자/이메일/CRM 연동)은 본 프로젝트 범위 밖이며, 시스템은 메시지·액션 결정까지를 책임진다고 가정한다.
- 외부 LLM 사용은 선택 사항이며, 평가·시연은 시뮬레이션 모드로도 완전하게 가능하다고 가정한다.
- 본 프로젝트는 학습 목적의 모의 프로젝트로, 실제 고객 PII 처리·운영 배포·법적 컴플라이언스는 범위 밖이며 공개 데이터만 사용한다고 가정한다.
- 약 8주 기간과 소규모 팀(예: 4인)을 가정하며, 진행은 별도의 스프린트 반복 프레이밍 없이 단계(모듈) 순차 진행으로 한다.
